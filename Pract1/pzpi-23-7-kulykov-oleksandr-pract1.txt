2 ЗАВДАННЯ

Практичне заняття присвячене дослідженню одного з класичних шаблонів проєктування з каталогу «Банди чотирьох» (GoF). У ході роботи студент має всебічно проаналізувати обраний патерн: розібрати його внутрішню організацію, визначити типові сценарії застосування та виробити практичні навички побудови програмних систем із мінімальною міжкомпонентною залежністю на підставі усталених архітектурних підходів.
Результатом виконання завдання є презентація, що висвітлює такі складові: теоретичне підґрунтя та систематизацію патернів GoF за категоріями; детальний розгляд структури обраного патерна та ролей його учасників; схему взаємодії об'єктів; практичну реалізацію з розбором програмного коду; зіставлення переваг і обмежень патерна; а також аргументовані висновки щодо доцільності його впровадження в реальних програмних проєктах.
 
3 ОПИС ВИКОНАНОЇ РОБОТИ

3.1 Фундаментальні основи та класифікація патернів GoF
Шаблони проєктування - це стандартизовані способи розв'язання типових завдань проєктування програмного забезпечення. Базовим довідником у цій галузі є книга «Design Patterns: Elements of Reusable Object-Oriented Software», видана у 1994 році авторами Еріхом Гаммою, Річардом Гелмом, Ральфом Джонсоном та Джоном Вліссідесом (відомими як GoF). У праці описано 23 класичні патерни, що поділяються на три основні категорії: породжувальні (Creation), структурні (Structural) та поведінкові (Behavioral). Патерн «Фасад», який є предметом цього дослідження, відноситься до категорії структурних патернів, оскільки він визначає способи організації класів у більші та гнучкіші структури.

3.2 Концептуальне визначення та ідеологія шаблону Facade
Патерн «Фасад» є структурним патерном проєктування, основне призначення якого полягає у наданні спрощеного та уніфікованого інтерфейсу до складної підсистеми класів. Головна мета цього архітектурного рішення - приховати внутрішню складність реалізації від клієнтського коду. Подібно до фасаду реальної будівлі, який формує її зовнішній вигляд та приховує внутрішні інженерні комунікації, клас-фасад у програмному забезпеченні слугує єдиною точкою входу, що абстрагує клієнта від складної бізнес-логіки компонентів системи.

3.3 Аналіз архітектурних викликів та передумови впровадження
У процесі розробки складних програмних рішень часто виникає архітектурна проблема, відома як жорстка зв'язність. Без використання фасаду клієнтський код змушений безпосередньо залежати від великої кількості об'єктів підсистеми. 
Клієнт має самостійно ініціалізувати десятки класів, розуміти їхні внутрішні залежності та дотримуватися строгого порядку виклику методів. Це призводить до того, що будь-яка зміна у внутрішній реалізації підсистеми вимагає каскадних змін у клієнтському коді. 
Рішенням цієї проблеми є впровадження патерну «Фасад», який бере на себе відповідальність за координацію роботи дрібних підсистем, дозволяючи клієнту взаємодіяти лише з одним високорівневим об'єктом.

3.4 Поглиблений розбір компонентної моделі та взаємодії учасників
Організація патерну «Фасад» ґрунтується на взаємодії трьох основних учасників:
-	Фасад (Facade): Це головний компонент, який знає, які класи підсистеми відповідають за обробку конкретного запиту. Він не містить в собі складної бізнес-логіки, а лише координує (оркеструє) роботу інших об'єктів. Його завдання - перекласти запит клієнта на мову, зрозумілу внутрішнім компонентам.
-	Класи підсистеми (Subsystem Classes): Це набір об'єктів, які виконують реальну роботу. Вони є вузькоспеціалізованими (наприклад, окремо клас для оплати, окремо для перевірки складу). Ключовий аспект полягає в тому, що класи підсистеми нічого не знають про Фасад. Вони не мають зворотних посилань на нього, що дозволяє використовувати їх як незалежно, так і через різні фасади.
-	Клієнт (Client): Користувач системи, який викликає методи Фасаду. Клієнту достатньо знати сигнатуру одного високорівневого методу.
Коли Клієнт викликає метод фасаду, останній ініціює ланцюжок викликів всередині підсистеми. Наприклад, один виклик PlaceOrder() з боку клієнта може трансформуватися у п'ять внутрішніх викликів: Inventory.Check(), Payment.Charge(), Warehouse.Reserve(), Logistics.Ship(), Mail.Notify(). Фасад керує життєвим циклом об'єктів підсистеми, забезпечуючи їх правильну послідовність та обробку помилок на кожному етапі.

3.5 Практичний приклад застосування: E-commerce
Для демонстрації практичного застосування патерну обрано модель бекенду системи оформлення замовлень в інтернет-магазині (E-commerce). Дана підсистема є комплексною і складається з багатьох спеціалізованих модулів: InventorySystem (відповідає за перевірку наявності товару на складі), PaymentGateway (забезпечує обробку фінансових транзакцій), ShippingService (організовує логістику та формування накладних) та NotificationService (відповідає за відправку електронних чеків користувачу).

3.6 Практична апробація: розробка системи E-commerce на С#
Для вирішення проблеми жорсткої зв'язності у системі E-commerce реалізовано клас OrderFacade. Цей клас інкапсулює у собі правильну послідовність виклику методів складної підсистеми. Він містить метод PlaceOrder, який послідовно виконує всі необхідні кроки: перевіряє склад, проводить платіж, оформлює доставку та надсилає сповіщення. Завдяки цьому використання системи клієнтом кардинально спрощується. У класі Program клієнт замість складної багатоетапної ініціалізації створює єдиний екземпляр фасаду (shopApi) і виконує лише один виклик методу PlaceOrder, передаючи базові параметри замовлення.

3.7 Переваги та недоліки патерну «Фасад»
Впровадження патерну «Фасад» характеризується низкою суттєвих переваг. Найголовніша з них - забезпечення слабкої зв'язності (Loose Coupling) між компонентами. Також патерн значно спрощує використання складних публічних API для зовнішніх розробників. Однак, архітекторам слід враховувати головний недолік цього підходу: за умови надмірного розширення функціоналу, існує високий ризик перетворення класу-фасаду на антипатерн «God Object» (об'єкт-бог) - монолітний компонент, який намагається керувати абсолютно всіма процесами в системі, що унеможливлює його подальшу підтримку.

3.8 Порівняння з іншими патернами
Патерн «Фасад» доцільно порівнювати з іншими структурними та поведінковими патернами для чіткого розуміння його унікальності. 
Порівняно з патерном Adapter, відмінність полягає у меті: Фасад створює абсолютно новий спрощений інтерфейс для підсистеми, тоді як Адаптер підганяє вже існуючий (але несумісний) інтерфейс до вимог клієнта. 
При порівнянні з патерном Mediator, головна різниця полягає у напрямку взаємодії: Фасад діє як односторонній шлюз доступу до підсистем (підсистеми не знають про фасад), тоді як Медіатор централізує та керує двосторонньою комунікацією між самими підсистемами.
 
4 ВИСНОВКИ

У ході виконання практичного заняття №1 було досліджено структурний патерн проєктування «Фасад» з каталогу GoF. На основі проведеного аналізу та програмної реалізації можна зробити висновок, що патерн «Фасад» є фундаментальним архітектурним рішенням. Він не додає системі нового функціоналу, але виконує критично важливу роль організатора коду: він приховує внутрішню складність реалізації від клієнта та вирішує проблему жорсткої зв'язності. Практична реалізація на прикладі бекенду інтернет-магазину мовою C# наочно довела, що використання фасаду кардинально покращує чистоту коду, його читабельність та легкість подальшого супроводу.

28  // Клас-Фасад, що приховує складну логіку оформлення замовлення
29  public class OrderFacade {
30  private InventorySystem _inventory = new InventorySystem();
31  private PaymentGateway _payment = new PaymentGateway();
32  private ShippingService _shipping = new ShippingService();
33  private NotificationService _notification = new NotificationService();
34  
35  public bool PlaceOrder(string productId, decimal price, string email, string address) {
36    if (!_inventory.CheckStock(productId)) return false;
37  
38    if (_payment.ProcessPayment(email, price)) {
39      _shipping.ArrangeShipping(productId, address);
40      _notification.SendReceipt(email);
41      return true;
42    }
43    return false;
44  }
45  }
